什么叫「深度用 AI」?
不是背提示词。

很多人以为「学 AI」就是学写提示词、收藏一堆咒语模板。过去两年,业界对「怎么用好 AI」的理解已经悄悄爬了 四层楼——提示词只是地基。而这四层爬上去的方向,和「AI 怎么发展」完全同步;爬到顶,正好回答那个终极问题:未来人和 AI,到底谁操作、谁负责。

第一部分 · 用 AI 的四层楼

提示词 → 上下文 → 驾驭 → 闭环

每上一层,就问一遍同一个问题:「人的判断,到底在哪一步发生?」 楼越高,人越省事,但人要负责的东西越重要。

LEVEL 01

提示词工程

Prompt Engineering
你说什么
  • 最底层:把一句话说清楚——给例子、给格式、给角色
  • 天花板:它假设「模型给一次答案」就完事了
  • 光背提示词,你永远停在「把 AI 当高级搜索框」

最底层就是大家熟悉的:你怎么把一句话说清楚——给例子、给格式、给角色。这层重要,但有个天花板:它假设「模型给一次答案」就完事了。

你说一句、它回一句,你不满意再改一句。人在每一次对话前后都得盯着。这也是为什么光背提示词,你永远停在「用 AI 当高级搜索框」的阶段。

LEVEL 02

上下文工程

Context Engineering
它能看到什么
  • 答得好不好,取决于它手上有多少料,不只是你这句话
  • 重点从「怎么措辞」→「该喂哪些资料、文档、工具、历史」
  • 上下文是有限资源:要精准,不是塞得越多越好

往上一层,大家开始意识到:模型答得好不好,不只取决于你这句话,而取决于它手上有多少料。

重点从「怎么措辞」变成「该把哪些资料、文档、工具、历史喂进它的视野」。核心认知是:上下文是有限资源,不是越多越好,而要精准。到这层你已经不只是聊天,而是在给 AI 布置一个信息环境。

LEVEL 03

驾驭工程

Harness Engineering
它在什么环境里干活
  • AI 开始「动手」:调工具、改文件、跑代码
  • 失败从「答错了」变成「系统出问题」——不是知识问题
  • 给它搭安全环境:工具 / 记忆 / 反馈 / 护栏 / 人工审批
  • 斯坦福:同一模型,环境好坏,表现差 6 倍

再往上,发生了根本变化:AI 不再只是「读懂、回答」的推理者,它开始动手——调用工具、改文件、跑代码。失败的形式也变了:不是「答错了」,而是 agent 陷入无法恢复的状态、把一个工具错调十二次、或没有护栏导致线上事故。

这些是系统问题,不是知识问题。所以这层要给 AI 搭一个能安全干活的环境:工具、记忆、反馈、护栏和人工审批点。斯坦福发现:同一个模型,光靠周边设计做得好不好,表现能差出 6 倍——不是换更强的模型,而是把环境搭好。

LEVEL 04

闭环工程

Loop Engineering
让它自己转起来
  • 不再手动一步步提示,而是设计自动控制流:自己提示、自评、自决、重试 / 上报
  • 你定义的不是「走几步」,而是目标 + 停止条件
  • 2026.6 爆火,一帖 650 万浏览;雏形是 Ralph 的 while 循环
  • 坑:失败按钱算——薄弱验证会整晚烧 token(Uber 烧光预算)

最顶层是 2026 年 6 月突然火起来的词:闭环工程。你不再手动一步步提示 agent,而是设计一套自动控制流,让它自己提示自己、自评、自决、重试或上报——不用人守在每一步旁边。你定义的是一个目标 + 一个停止条件,循环自己去迭代。

但有个必须讲的坑:这是四层里第一次,失败的代价用钱来算。一个验证薄弱的循环,会整晚不停产出烂答案,每个都在烧 token。据说 Uber 就因此几个月烧光年度预算,后来给工程师定了每月 1500 美元上限。所以——没有停止条件和成本控制,这套东西就会崩;闭环一大半的工作,其实是确保它能停下来。

这是 AI 发展的必然

这不是四个技巧,而是 AI 能力长大的四个阶段——从「每句话都我说」,到「我搭好环境让它干」,再到最后 「我只定目标和验收标准」。

「深度用 AI」的真正含义,不是提示词写得多花哨,而是你站在第几层楼。记住这句——它是最后一部分的伏笔。

落地补充 · 通用 HARNESS

你不用造车,你只要会开

「驾驭」和「闭环」这两层,普通人根本不用自己搭。现成、通用的「AI 干活平台」有个统一名字,叫 harness(AI 的驾驶舱)。四层楼在讲一辆车怎么造;而 Claude Code、Codex 就是造好的整车——你只要学会怎么开。

它跟「聊天框」差在哪?

一句话:它会动手,不只是动嘴。

聊天框 · 只动嘴
Harness · 会动手

官方描述很到位:它读完你的整个项目,规划跨多个文件的方案,执行修改、运行测试、失败自己迭代——而你,定义目标、验收结果,而不是指挥每一步。用一个通用 harness,你其实已经站在第三、第四层楼上了,只是这栋楼不是你盖的。

它不只是给程序员的

请把「coding 工具」这个标签摘掉,换成——一个能在你电脑里替你动手干活的 AI 助手。它能操作文件、整理文档、处理表格、跑浏览器验证。

按公司模板生成 Word 拍收据算 AA 账 翻译长报告 整理表格 跑浏览器验证 → 都不是「编程」

01

拆:把大而模糊的目标,拆成小而明确的活

  • 「帮我做个宣传视频」是一坨模糊,只能还你平庸的平均答案
  • 先定脚本 → 分镜 → 每镜画面文案 → 时长节奏 → 渲染合成
  • 拆得越细,做得越准,随时能在某块喊停微调

「帮我做个宣传视频」是一坨模糊的东西,Codex 只能还你一个平庸的平均答案。工程思维的第一反应是把它拆开。

你拆得越细,它做得越准。每一小块都清晰、可检查,你随时能在某块喊停微调,而不是等它把一整个烂视频做完才发现方向错了。一个片子毁,往往毁在最开始那个模糊的 brief 上。

02

说清「什么算做完」——模糊进,模糊出

  • 多长?什么风格?给谁看?什么调性?横屏还是竖屏?
  • 杀手锏:别用嘴描述,用东西喂它
  • 把参考视频、参考图、过去最好的片子丢给它:「我要这个味道」

不能只说「做得好看点」,要说清多长、什么风格、给谁看、什么调性、横竖屏、参考谁——你越能说清「什么样算对」,它越不会跑偏。

杀手锏:别用嘴描述,用东西喂它。直接把参考视频、参考图、你们过去做得最好的那个片子丢给它。给它一个「好」的样板,比写一百字形容词都管用。

03

给它「料」和「工作间」

  • 做视频前把品牌色、logo、字体、口播稿、素材、旧成片都给齐
  • 整理进一个文件夹交给它,它才能照着你的家底干活
  • 上下文越齐整,它越像你团队里的人,而不是外包路人

Codex 强在能在你的文件夹里动手。做视频前把该给的都给齐:品牌色、logo、字体、口播稿、素材、以前的成片,整理进一个文件夹交给它,它才能照着你的家底干活,而不是凭空瞎编一个跟你品牌八竿子打不着的东西。

★ 做视频的人重点听
04

把视频当成「可以重新生成的系统」

  • 普通人一帧帧手工剪,改一个字要回去重剪——这是手工作坊
  • 让视频本身变成一套「配置 / 代码」,可以重新生成
  • 改一个参数整片重出 · 一次定义批量三尺寸 · 中英 / A-B 一次全批
  • 手工的人做「一个」视频,工程思维的人做「一台产视频的机器」

普通人做视频是在剪辑软件里一帧帧手工剪,改一个字要回去重剪,做十个版本要重复十遍——这是手工作坊。Codex 会写代码,意味着你可以让视频本身变成一套「配置」或「代码」。

客户说「字幕都多停两秒」——改一个参数整片自动重出;要抖音竖版、B站横版、朋友圈方形——定义一次批量三个;要十个 A/B 标题或中英双语——一次全批。你可以用 Remotion 这类「用代码做视频」的工具,搭一个改参数就重出的模板,试一次就回不去了。

05

验收 + 存档:把一次成功,变成永久能力

  • 节奏对不对、信息准不准、有没有低级错误——你懂宣传,AI 不知道
  • 把「你怎么一步步指挥它」整理成说明,或做成可复用的模板 / skill
  • 下次做同类片子直接调出来,五分钟起步,不用从零教

它交活你必须会看:节奏对不对、信息准不准、有没有低级错误。这是你不可替代的价值——你懂宣传,你知道什么打动人,AI 不知道。

更关键的是存档:把「你怎么一步步指挥它做出来」整理成说明,或做成可复用的模板 / skill。下次直接调出来,五分钟起步。

〔 演示位 · 「改一个参数,整片重出」或「一次定义,批量三个尺寸」 〕

会开车的人,眼睛看的是路,不是方向盘。

技巧会过时,思维不会。拆解、定义完成、备好上下文、把事情系统化、验收存档——这五个动作跟「视频」没有半点绑定,今天做宣传片,明天做报告、搭工具、处理表格,一模一样地好使。

第三部分 · SKILL

把一件事,打包下来

Skill 就是把「做某件事的完整方法」打包成一个文件夹,以后 AI 一遇到这类事就自动照做。判断信号:如果你在反复地、跨很多次对话敲同一段提示词——那就该把它做成一个 skill 了。

判断信号 → 打包动作

发现自己在反复、跨很多次对话敲同一段提示词——那就是该打包的信号。

① 借别人的

Taste、UI/UX Pro Max、Vercel 审查 skill——别人打包好的,拿来即用,站在巨人肩膀上。

② 沉淀你自己的 ★

把「只有你会、每次都要手把手教」的独门流程打包——你把经验变成 AI 可永久复用的能力。教一次,用一辈子。

Skill 是你个人经验的「存钱罐」——每沉淀一个,就把自己从一件重复劳动里彻底解放一次。

第三篇 · PART THREE
UI 设计 & GitHub
3.1 · 让作品去掉「AI 味」

为什么大家做的 UI 长得一模一样?

居中大标题、三张并排卡片、渐变紫按钮、假数据统计、一排图标——社区管这叫 slop(设计泔水)。根源是模型会向训练数据的平均值回归。所以核心只有一句:与其做完再修,不如一开始就给它一套更高级的参考,让泔水根本生不出来。

✕AI slop · 向平均值回归

居中大标题 · 三张并排卡片 · 渐变紫按钮 · 假数据统计 · 一排图标。

✓更高的起点 · 有观点、有层次

不对称编辑式布局 · 克制的单色 + 一个强调色 · 真实的层级。

接下来三种打法,本质是三种「换参考」——与其做完再修,不如一开始就给它更高级的参考。

打法 01

design.md 复刻流派

用文档换参考
  • 给 AI 一份「设计说明书」,几 KB 一次性加载,最省最稳
  • anydesign:喂图 / 网址 / Figma,生成结构化 design.md
  • 最狠是不瞎编——每个推断带置信度,颜色全是真实 hex
  • 也可把自己做过的项目沉淀成 design.md,下次复用

为什么是 Markdown?上下文窗口有限。一份几 KB 的 design.md 可以一次性当背景加载;每次去读庞大 Figma 或反复查询,每轮迭代都在烧 token。

anydesign 给它图/网址/Figma,就生成结构化 design.md,配色 token、组件清单、复刻笔记都列好。它不瞎编:每个推断带置信度,颜色全是真实 hex。扒过 Vercel 官网,提取 800 多个样式属性一个没编。

打法 02

skills 审查 + 渲染验证闭环

用规则加护栏
  • 审查 skill 分两类,别装混了会打架
  • 视觉品味类:UI/UX Pro Max(~10万 star)、Taste、Impeccable
  • 质量门禁类:Vercel web-design-guidelines(焦点 / 触控 / 语义 / 键盘)
  • 最佳组合:一个视觉类 + 一个技术类
  • 渲染验证:用 Playwright 让 AI 自己截图、自查、自修

第一类管「好不好看」:UI/UX Pro Max 是可搜索知识库;Taste 给你变化度、动效、密度三个旋钮;Impeccable 让你和 AI 共享一套设计词汇。第二类管「对不对」:Vercel 的 web-design-guidelines 像给 UI 装了体检 linter,只挑错不生成。

记住:一个视觉类 + 一个技术类。千万别同时装两个视觉类,它们会互相下矛盾指令。再加渲染验证闭环——让 AI 自己截图自查自修,输出稳定性会明显上台阶。

打法 03

图生代码 / 参考图工作流

用图片当参照
  • 门槛最低,完全不需要你懂设计
  • 第一步:用图片生成先「画」出界面参考图
  • 第二步:把参考图 + 现有代码一起丢给 AI,分析风格再套用
  • 绕过「不知道怎么用文字描述好看」这个最大障碍

先用 AI 生成设计参考图,再让 AI 照着复刻成代码。第一步用图片生成画出界面参考图,你只要描述氛围、风格、大概布局;第二步把参考图连同现有代码丢给 AI,让它分析风格再套用。

它妙在绕过了「你不知道怎么用文字描述好看」这个最大障碍——你不用会讲设计术语,只要能指着一张图说「我要这个感觉」。对完全不懂 UI 的同事,这是最快出成果的路。

!

安全提醒:Snyk 研究发现约 13% 被测 skill 含严重安全缺陷,有的会偷偷窃取凭证。skill 本质是带脚本的文件夹,恶意的能藏后门。准则:非官方来源的 skill,装之前先把它的 SKILL.md 读一遍。

三种打法,一个内核:别让 AI 回归平庸,给它一个更高的起点。 可以叠着用——先出参考图定方向,沉淀成 design.md,再用 skills 审查兜底。
